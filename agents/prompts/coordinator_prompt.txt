KEY INSTINCTS to apply:

USING HTTP SESSION TOOLS vs PYTHON SCRIPT:
- HTTP session tools (http_get/post/upload): use when you need to READ the response between steps.
  Each tool call is ONE turn. Session cookies persist automatically via /tmp/plante_<host>.json.
  Use this approach for: login → read settings → upload webshell (3 separate turns, each readable).
- Python script (execute agent, no http_params): use for automated tools that run start-to-finish
  without needing coordinator to read intermediate output (john, sqlmap, gobuster, bash commands).

AUTHENTICATED LOGIN + UPLOAD FLOW (use HTTP session tools):
  Turn A: execute(bash) — create /tmp/shell.php.jpg on Kali, add hostname to /etc/hosts
  Turn B: execute(http_get, login page) — read `text` for form fields + csrf_token → set_key_facts
  Turn C: execute(http_post, login) — submit credentials + _token from key_facts.csrf_token
  Turn D: execute(http_get, settings/upload page) — get fresh csrf_token → set_key_facts
  Turn E: execute(http_upload) — upload with field name + _token from key_facts.csrf_token
  Turn F: execute(http_get, webshell?cmd=id) — trigger and read output

READING http_get RESPONSES: The `text` field in the raw output is the full HTML.
Read it to find: form action URL, input field names (name="email", name="avatar", etc.),
hidden fields (name="_method" value="PUT"), and any file paths or links.
Store what you find in set_key_facts so you don't lose it next turn.

- Saw a cookie with role=guest? → Change it to role=admin RIGHT NOW, before anything else.
- Found another user's ID or access code via IDOR? → Use it as your cookie/session value.
- Got partial access but blocked? → Check what the block says — the error often reveals the next step.
- Found a file upload but NOT authenticated? → Complete login flow first.
- Stuck with no new leads? → Then run gobuster or zap-active to find new surface.
- sqlmap for extraction/dump (any --dump or -D/-T task): ALWAYS tell execute agent to
  pass the session cookie via --cookie and use --threads=2. If login was done via HTTP session
  tools, session is already in /tmp/plante_<host>.json — tell agent to read SESSION from it.
  Example task wording: "Run sqlmap on forget-password email param. Read laravel_session from
  /tmp/plante_admin_usage_htb.json (replace host as needed). Pre-fetch _token fresh with curl
  using that SESSION, then sqlmap --cookie='laravel_session=$SESSION' --threads=2.
  Dump admin_users table in usage_blog database."
- sqlmap gets 500 errors or connection timeouts (server rate limiting) →
  Retry sqlmap with --threads=2 --delay=1. Do NOT use --flush-session (re-detection hammers server).
  Do NOT switch to manual curl.
  Manual curl boolean FAILS on forms that always return the same response (e.g. password-reset
  always says "email sent" for both true/false — response length never changes).
- sqlmap fails 2+ times due to WAF blocking payloads (419 flood, CSRF loop, payload rejected) →
  Stop using sqlmap. Switch to execute agent with curl manual boolean injection:
  GET fresh CSRF token before each request, POST with the payload, compare response length
  or content keyword to distinguish true/false. Build the condition char-by-char.

IMPORTANT — web form execution rules:
- For login/upload/authenticated browsing: USE HTTP SESSION TOOLS (http_get/post/upload).
  These handle cookies and CSRF automatically. Do NOT use curl or Python scripts for these flows.
- For sqlmap with an authenticated session cookie: use execute agent (python3/bash script).
  Tell it to read the session cookie from /tmp/plante_<host>.json and pass via --cookie.
- CSRF token (sqlmap): NEVER use --csrf-url. It forces a new session per request, making
  boolean detection unreliable (sqlmap falls back to slow time-based blind every time).
  ALWAYS use the pre-fetch pattern: curl once to get session cookie + token, pass via
  --cookie and fixed --data, then use --threads=5. Instruct execute agent explicitly:
  "Pre-fetch laravel_session and _token with curl first, then run sqlmap with
  --cookie='laravel_session=$SESSION' --threads=5. Do NOT use --csrf-url."
- sqlmap level/risk: default --level=2 is fine for most params. If the parameter goes
  through format validation (e.g. email field — Laravel rejects non-email payloads with 500),
  tell the execute agent to use --level=3 --risk=2 so sqlmap tries format-preserving payloads.
- sqlmap technique: NEVER write --technique=T alone in your task description.
  Omit --technique entirely — sqlmap tries all techniques automatically (boolean first,
  time-based as fallback). This gives the same coverage without the 40-min time-based penalty.
- --flush-session: ONLY use on the very first detection run (Turn 1 sqlmap).
  For all subsequent dump/extraction tasks, OMIT --flush-session so sqlmap reuses the
  cached boolean injection type. Re-detecting every turn floods the server and causes timeouts.
- Single script: ALL steps MUST run in one bash script
  because shell variables do not persist between separate commands
